## Design choices worth flagging

- **Single `config` JSON column** rather than separate `vernacular` + `tracker` columns. Keeps schema flat and lets future config sub-objects (notifications, theme preferences, etc.) land without migrations. Trade-off: harder to index/query on specific config keys; can add Postgres expression indexes if a use case emerges.
- **Validation is loose.** I deliberately did NOT validate vernacular's individual keys (coordinator, worker, etc.) — that would tie the schema to a fixed vocabulary, which fights the whole "user-defined" point. The `Vernacular` module in gte-P2 will fall back to gas-town defaults for missing keys rather than reject.
- **Tracker enum hardcoded as `~w(none jira linear github)`** in `Workspace.valid_tracker_types/0`. When we add new adapters (Notion, etc.), this list grows. Could move to config later if it becomes annoying.
- **`require_atomic? false` on `:update`**. Custom changes that read & validate the JSON aren't atomic-safe (they need to inspect the value before deciding). Ash 3 requires opting out explicitly. No performance concern at our scale.
- **No `has_many :issues, :convoys` yet.** Issues (gte-002) and Convoys (gte-004) don't exist. Relationships will land in those beads.
- **String keys in `config`.** JSON in Postgres comes back as `%{"key" => value}`, not `%{key: value}`. I went all-strings rather than atomize-then-stringify-on-write. Slightly more verbose at the application layer but avoids surprise data corruption from `String.to_existing_atom/1` traps.
- **UUID v7 PK** (Ash 3 default with `uuid_v7_primary_key`). Time-ordered, good for indexing. Phoenix scaffold ships the `uuid_generate_v7()` function in the extensions migration from gte-001.

## What I punted on (with reasons)

1. **`Vernacular` module (`Vernacular.label/1`, `Vernacular.alias_resolve/1`, etc.)** — that's gte-P2's bead.
2. **`Tracker` behaviour and adapters** — gte-019 and gte-029.
3. **Process-dictionary cache of the current workspace** — gte-P2 will add this so `Vernacular.label/1` doesn't hit the DB on every call.
4. **`bd2 workspace ...` CLI commands** — gte-006 (CLI escript) will expose workspace management; for now, workspaces are created via Ash directly or by editing seeds.exs.
5. **LiveView settings page for editing the config JSON** — gte-P4.

## What I noticed worth improving separately

- **The `config` description in the resource module is multi-line.** Looks fine in IDE but the line break inside the heredoc could be tighter. Cosmetic.
- **The `change/2` callback in `ValidateConfig`** has a slightly verbose `then/then` chain. Could refactor with `Enum.reduce` over a list of validators, but the current form is readable and only ~80 lines.
- **No test for the seeds script itself.** I tested idempotency manually (ran twice, count stayed at 1). Could codify in an integration test that boots a fresh test DB, runs seeds, asserts state. Low ROI right now.
